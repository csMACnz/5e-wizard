## Bug #1 — Application depends on external Google Fonts/Icons CDN

**Severity:** Medium  
**Category:** Performance / UX / Reliability

### Description

`src/CharacterWizard.Client/wwwroot/index.html` loads fonts and Material Icons from Google's external CDN:

```html
<link rel="stylesheet" href="https://fonts.googleapis.com/css?family=Roboto:300,400,500,700&display=swap" />
<link rel="stylesheet" href="https://fonts.googleapis.com/icon?family=Material+Icons" />
```

These resources **fail to load** in offline environments or networks that block Google services.

### Steps to Reproduce

1. Open the app in an offline or network-restricted environment
2. Observe the visual degradation — all icon buttons show broken placeholder boxes

### Screenshot

![Homepage with broken Material Icons](https://raw.githubusercontent.com/csMACnz/5e-wizard/copilot/playwright-bug-reports/docs/playtest-screenshots/bug1-google-fonts-failure.png)

*Screenshot taken in a sandboxed environment with no internet access to Google's CDN. All icon buttons (dice, upload, navigation, etc.) show blank/broken squares.*

### Suggested Fix

- Use MudBlazor's built-in SVG icons () instead of the external icon font CDN
- Bundle the Roboto font locally (e.g., via  npm package) and serve from

## Bug #2 — Print Sheet renders blank when accessed without a valid session

**Severity:** Medium  
**Category:** UX / Navigation

### Description

Navigating directly to `/print` without completing the wizard (or after a session expires) shows a fully blank D&D character sheet with "—" in every field — no error message, no redirect, no guidance.

### Steps to Reproduce

1. Open the app in a fresh browser tab
2. Navigate directly to `/print`

### Expected Behavior

Redirect to the home page / wizard, or display a clear "No character loaded" message.

### Screenshot

![Blank print sheet](https://raw.githubusercontent.com/csMACnz/5e-wizard/copilot/playwright-bug-reports/docs/playtest-screenshots/bug2-print-sheet-blank.png)

*Every character field shows "—". No guidance is given to the user.*

### Affected File

`src/CharacterWizard.Client/Pages/PrintSheet.razor` — add a guard checking if `State.Character` has meaningful data (e.g., non-null Name or Levels) before rendering the sheet.

## Bug #3 — Standard Array: already-assigned scores remain selectable for other abilities

**Severity:** Medium  
**Category:** UX / Wizard

### Description

When using **Standard Array** on Step 2, the six values [15, 14, 13, 12, 10, 8] must each be assigned exactly once. However, assigning 15 to STR does not remove 15 from the DEX dropdown — the value remains selectable, allowing accidental or deliberate duplicate assignments.

### Steps to Reproduce

1. Start a new character, select **Standard Array** on Step 1
2. Go to Step 2 (Ability Scores)
3. Assign **15** to STR
4. Open the **DEX** dropdown — 15 is still listed

### Screenshot

![DEX dropdown still shows 15 after STR=15](https://raw.githubusercontent.com/csMACnz/5e-wizard/copilot/playwright-bug-reports/docs/playtest-screenshots/bug3-standard-array-duplicates.png)

*After assigning 15 to STR, the DEX dropdown still shows "15 (mod 2)" as an option.*

### Note

The "Next" button IS correctly disabled when duplicates exist, but the UI gives no indication of which values are already used. Users must guess why they are blocked from proceeding.

### Affected File

`src/CharacterWizard.Client/Pages/WizardSteps/WizardStepAbilities.razor` — filter already-selected values from each ability's dropdown options.

## Bug #5 — ASI Split (+1/+1) allows selecting the same ability for both improvement slots

**Severity:** High  
**Category:** Rules Compliance / Wizard

### Description

On Step 5 (Level Features), when choosing the "+1/+1 to two abilities" ASI option, the same ability (e.g., STR) can be selected for both "First ability (+1)" and "Second ability (+1)". D&D 5e rules require the split to apply to **two different** abilities.

### Steps to Reproduce

1. Create a Fighter character at **Level 4** (Fighter gets ASI at level 4)
2. Proceed to Step 5 (Level Features)
3. Select "+1 / +1 to two abilities"
4. Set "First ability (+1)" = **STR**
5. Open "Second ability (+1)" — STR is still listed
6. Select **STR** again

### Expected Behavior

Once STR is selected for the first slot, it should be **unavailable** in the second slot.

### Code Evidence

`src/CharacterWizard.Shared/Validation/LevelFeatureValidator.cs` — the `Validate()` method only warns for missing ASI choices and errors on invalid feat IDs. There is no check that `choice.AbilityOne != choice.AbilityTwo` when the mode is "split".

### Screenshot

![Step 5 ASI panel for Fighter Level 4](https://raw.githubusercontent.com/csMACnz/5e-wizard/copilot/playwright-bug-reports/docs/playtest-screenshots/bug5-step5-next-no-asi-warning.png)

### Affected Files

- `src/CharacterWizard.Shared/Validation/LevelFeatureValidator.cs` — add validation: `AbilityOne != AbilityTwo` for split mode
- `src/CharacterWizard.Client/Pages/WizardSteps/WizardStepFeatures.razor` — filter the already-selected first ability from the second ability dropdown

## Bug #6 — Step 5 (Level Features) does not block advancing with unresolved ASI choices

**Severity:** Medium  
**Category:** UX / Wizard / Rules Compliance

### Description

When a character has an unresolved ASI (Ability Score Improvement) at Step 5, the wizard only generates a **warning** (non-blocking), leaving the "Next" button enabled. Users can skip ASI selection and proceed to Step 6, resulting in an incomplete character that fails final validation at the Review step.

### Steps to Reproduce

1. Create a Fighter character at **Level 4**
2. Proceed to Step 5 (Level Features)
3. Do **not** make any ASI selection
4. Click **Next** — the wizard advances to Step 6 without error

### Code Evidence

`src/CharacterWizard.Client/Services/WizardStepValidator.cs` case 4:

```csharp
// Features step: warn for incomplete ASI choices (non-blocking)
...
result.Warnings.Add($"WARN_ASI_INCOMPLETE: ASI at ...");
```

The comment "non-blocking" indicates this is intentional design, but it creates a poor UX where incomplete choices are only caught at the review stage.

### Screenshot

![Step 5 with Next button enabled and no ASI selected](https://raw.githubusercontent.com/csMACnz/5e-wizard/copilot/playwright-bug-reports/docs/playtest-screenshots/bug5-step5-next-no-asi-warning.png)

*The Next button remains enabled even with an unresolved ASI choice.*

### Suggested Fix

Change the `WARN_ASI_INCOMPLETE` result from a warning to an error in `WizardStepValidator` case 4, so the Next button is disabled until all ASI choices are made.
